## 构建 passwdpm-web：服务端 GPG/SM2 加解密密码管理器

完成了一个完整的全栈密码管理工具（Python 后端 + 原生前端）。核心特性：加解密全部在服务器完成、每条记录可选 GPG 或 SM2、密码查看/新增/修改/删除、修改记录（审计日志）。

### 关键技术决策（踩坑记录，重要）
- **运行环境**：使用托管 Python `C:\Users\12580\.workbuddy\binaries\python\versions\3.13.12`，venv 在 `...\envs\passwdpm`。后端依赖装在该隔离 venv。
- **pgpy 在 Python 3.13 不可用（`imghdr` 被移除）**：在 `backend/app/crypto/gpg_crypto.py` 顶部注入 `imghdr` 垫片（`sys.modules['imghdr']`）后再 import pgpy。常量名是 `SymmetricKeyAlgorithm`（非 `SymAlgorithm`）。
- **gmssl SM2 公钥推导**：`CryptSM2` 不会自动由私钥生成公钥。用 `c._kg(int(priv,16), c.ecc_table['g'])` 算 Q=priv*G。加密结果 base64 存储。
- **passlib + bcrypt 不兼容**：passlib 1.7.4 与 bcrypt>=4.1 冲突（72 字节检测报错）。必须固定 `bcrypt==4.0.1`（已写进 requirements.txt）。
- **run.py 入口**：用绝对导入 `from app.config import ...`，从 `backend/` 目录运行 `python run.py`（不能用相对导入，否则 "attempted relative import with no known parent package"）。
- **FastAPI 0.139 的 include_router**：被包成 `_IncludedRouter`，`app.routes` 不直接展开 API 路径；需用 TestClient 或真实请求验证，不能只看 routes 列表。
- **密钥/数据库**：首次启动 `seed.seed()` 自动建表、建管理员（env ADMIN_USERNAME/ADMIN_PASSWORD，默认 admin/admin123）、生成 gpg+sm2 密钥对存入 SQLite（`backend/data/passwdpm.db`）。JWT 密钥写入 `backend/.secret_key` 持久化（已 gitignore）。

## Docker 镜像重建与端到端验证

**根因：docker 构建失败 → daocloud 公共镜像源 401 Unauthorized**
- 现象：`docker build` 报 `failed to resolve source metadata for docker.io/library/python:3.13-slim: unexpected status from HEAD request to https://docker.m.daocloud.io/...: 401 Unauthorized`。
- 根因：本机 `~/.docker/daemon.json` 配置了 `"registry-mirrors": ["https://docker.m.daocloud.io"]`，该公共加速器已要求鉴权，不登录返回 401。
- 处理：备份后移除 `registry-mirrors` 字段，让 Docker Desktop 直连 docker.io。原配置已备份为 `daemon.json.bak.YYYYMMDD_HHMMSS`。
- 重启 Docker Desktop 时沙箱里 `osascript quit` 报「权限违例 -10004」（无障碍权限），但守护进程本来就好，直接 `open -a Docker` 后 daemon.json 新配置自动生效。

**架构陷阱：arm64 主机 vs 目标 amd64 服务器**
- 现状：本机 macOS arm64，Docker Desktop 4.80，默认 build 产出 arm64 镜像。
- 目标：离线部署服务器是 x86_64 linux，必须显式 `--platform linux/amd64`，否则镜像无法在目标服务器运行。
- QEMU 模拟可用：`docker run --rm --platform linux/amd64 alpine:latest uname -m` 输出 `x86_64`，可放心跑 amd64 容器做端到端验证。
- 现有 `build_image.sh` / `docker-compose.yml` 都未指定平台 → **生产构建必须传 `--platform linux/amd64`**。

**沙箱 + curl 的坑：multipart form 上传被拦截**
- 现象：主机上 `curl -F` 跑两次后第二次报 `(eval):N: failed to change group ID: operation not permitted`，curl 直接被打死（不是 curl 本身问题，是 WorkBuddy 沙箱拦截 setgid）。
- 解法 1（推荐）：`docker cp script.py container:/tmp/ && docker exec container python3 /tmp/script.py` —— 在容器内跑测试，绕开主机 sandbox。
- 解法 2：在容器里用纯 Python `urllib` 拼 multipart（不要依赖 curl）。slim 镜像只有 python3，没有 curl。
- **前 11 项测试都通过后才出现这个错误，是因为 curl 在同一会话多次 exec 后随机触发 sandbox 限制，不是确定性问题。**

## 本会话第二轮改动（用户三项需求）

### Bug 1：导出报 `body stream already read`
- 根因：`app.js#apiBlob` 对**所有响应**都先 `await res.json()`；JSON 解析失败会被 try/catch 吞掉，但 stream 已被消耗；后续 `await res.blob()` 就抛 "body stream already read"。
- 修复：先看 Content-Type。JSON 才 parse error 体；其他（如公钥文本、二进制）直接走 blob 路径。
- **另踩**：`exportOrgKey` 用了 `?type=` 但后端 Query 参数名是 `kind`（之前测试时 `kind=public` 巧合成功，实际 `private` 请求会回落到默认 public —— 静默 bug）。

### 关键经验总结（值得写入 skill）
1. **fetch 响应体只能消费一次**——分 JSON/Blob 路径必须先看 Content-Type，不要对所有响应都 `await res.json()` 试探。详见《再谈 fetch body stream》式陷阱。
2. **后端 Query 参数名与前端参数对齐**——`?type=` vs `?kind=` 这种 silent 默认值 bug 是非常隐蔽的回归点。修 bug 顺带把整个项目所有 query 参数名都校对一遍。
3. **PUT/PATCH 路径也要和 GET 路径走相同的解密链路**——同一字段的查询和修改如果走不同代码分支容易 500。把"能否解密"判断抽成公共函数。
